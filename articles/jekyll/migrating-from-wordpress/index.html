Migrating from Wordpress

The Import Process

I have been using Wordpress for about 5 years now and while it was easy to get running it wasn't the best platform for me. While deciding if I would leave Wordpress behind I came up with a few requirements for the new site. First I wanted the pages to load as fast as possible and you cannot get any faster than loading up static html pages. The second requirement was related to security, I wanted the site to be as secure as possible, cutting out the database layer and any dynamic functionality will increase security quite a bit. Once I made the decision to make the switch it was time to investigate how I would get my content migrated over to the new platform. If you do not plan to import your posts from Wordpress you can feel free to skip this article, if you are importing from another source you will have to ask Google how it should be done. If you are not going to be importing any data you can start to create your posts now.

When I first started migrating to Jekyll I was following this walk-through. I was not aware that Jekyll had some migrators built-in one of which is for Wordpress. As I was not aware that these migrators were available I personally did not use them therefore cannot document them here. However the process is the same if you decide to use the built-in migrators the only thing you should have to change is the path to the file in the command we use. There are two ways in which you can import your content from Wordpress, the first is by using the database and the second is by using an export file from Wordpress. I will be covering the method that uses the database for the content.

After I made the decision to use the database for my content I found that the database on my web host does not allow for remote connections, I then had to export my entire database and import it to a local instance of MySQL that I have on my development machine. During this process I also had to setup the database user and set the scheme privledges for the database user. Once I had the data in place it was time to actually get the script used for the import and put that in place.

I created a directory in my site project named _lib/ and created a wordpress.rb file. I took the source from the walk-through that I was following and modified it a bit to meet my needs. First I changed the files from being .markdown to .md for shorter filenames. I then changed the way categories work. I have and always will use a single word for a category, I will not be having categories which use multiple words so I removed the title, slug and autoslug from being generated during the import. I simply wanted a list of category names to be put into an array in the YAML Front Matter for each post, I also removed some data that I found to be not needed such as the wordpress url and post id. Below is the full source for my importer script. Note that I choose to name this directory with an underscore so that the import script would not be put in the site generation output, I only wanted my site to consist of static files and not any script files.

%w(rubygems sequel fileutils yaml active_support/inflector).each{|g| require g}

require File.join(File.dirname(__FILE__), "downmark_it.rb")

module WordPress
    def self.import(database, user, password, table_prefix = "wp", host = 'localhost')
        db = Sequel.mysql(database, :user => user, :password => password, :host => host, :encoding => 'utf8')

        %w(_posts _drafts images/posts/featured).each{|folder| FileUtils.mkdir_p folder}

        query = <<-EOS

      SELECT    post_title, post_name, post_date, post_content, post_excerpt, ID, guid, post_status, post_type, post_status,
             (  SELECT  guid
                FROM    #{table_prefix}_posts
        WHERE   ID = (  SELECT  meta_value
                                FROM    #{table_prefix}_postmeta
        WHERE   post_id = post.ID AND meta_key = "_thumbnail_id") ) AS post_image

      FROM #{table_prefix}_posts AS post
      WHERE  post_type = 'post'

        EOS

        categories_and_tags_query = <<-EOS

      SELECT          t.taxonomy, term.name, term.slug
      FROM          #{table_prefix}_term_relationships AS tr
      INNER JOIN    #{table_prefix}_term_taxonomy AS t ON t.term_taxonomy_id = tr.term_taxonomy_id
      INNER JOIN    #{table_prefix}_terms AS term ON term.term_id = t.term_id
      WHERE         tr.object_id = %d
      ORDER BY    tr.term_order

        EOS

        db[query].each do |post|
            title      = post[:post_title]
            slug       = post[:post_name]
            date       = post[:post_date]
            content    = DownmarkIt.to_markdown post[:post_content]
            status     = post[:post_status]
            name       = "%02d-%02d-%02d-%s.md" % [date.year, date.month, date.day, slug]
            image      = File.basename(post[:post_image]) rescue ""
            categories = []
            post_tags  = []

            puts title

            `wget -O "images/posts/featured/#{image}" "#{post[:post_image]}"` unless File::exists?("images/posts/featured/#{image}") || post[:post_image].nil?

            db[categories_and_tags_query % post[:ID]].each do |category_or_tag|
                eval(category_or_tag[:taxonomy].pluralize) << category_or_tag[:name]




            end

            data = {
                'layout'        => 'post',
                'title'         => title.to_s,
                'is_review'     => post[:post_title].start_with?("Review: "),
                'excerpt'       => post[:post_excerpt].to_s,
                'image'         => image,
                'tags'          => post_tags,

                'categories'    => categories

            }.delete_if { |k,v| v.nil? || v == ''}.to_yaml

            File.open("#{status == 'publish' ? '_posts' : '_drafts'}/#{name}", "w") do |f|
              f.puts data
              f.puts "---"
              f.puts content
            end
        end
    end
end

One thing to note about the code above is that I am using Markdown and the import process will need a way to convert HTML to markdown. To do this it uses the downmark_it ruby module, which you can find here. You will have to put this in the _lib directory along side the wordpress.rb file you created. I have also included it in the archive attached to this article.

When you are going through this process feel free to investigate every method for importing and choose the one that best suits your needs, there is not going to be one uniform way to do it for everyone. You might even find that you have to customize the importing script to get any specific data you need. I would also urge you to try out the build in migrator for Wordpress, I might even go backup my current files and try giving it a run myself to see if and how it could benefit me.

NOTE: If you choose to import your content from your database you must have the MySQL Rugy Gem installed, I am on Mac OS X and initially I had problems getting this to install and work properly. I fought with Ruby and the gems for what seemed like hours before digging deep into Google and finding a solution. If you cannot get the mysql gem to install I would suggest that you uninstall the mysql gem and then install the ruby-mysql gem. Once I installed this everything went so smooth. What I read on Google was that the mysql gem was no longer being actively maintained and that one should now use the ruby-mysql gem. I spent hours figuring this out so don't let it bite you as well.

To run the import process you will need to load up your terminal and make sure you are in your site root directory then execute the following command substituting your database credentials where required.

ruby -r './_lib/wordpress.rb' -e 'WordPress::import("codeb2_wpb", "codeb2_wpb", "NiX19E", "wp")'

When you exectute this command the process should take off reading your database and creating the necessary files it should also download any featured images associated with the posts. Your terminal output should look similar to the output below.

I have limited the output above as it is only to be an example of what you should see when you run the import process. I had a lot more data in my terminal window so don't be shocked if you see a lot of data fly through the screen. You will notice that in the above output if the post has a featured image the script pulls that down and saves it to the sites images directory, if there is no featured post it will only pull down the post data and create the post file.

Once I had all of my files in place after the import I noticed that I would have to do quite a bit of formatting to these files. My Wordpress install did not use Markup for the posts content, if it had migration would probably have been easier. While the posts content should work out of the box after you import I didn't like how it was all HTML content in my markdown files. I started to re-format these posts using Markdown but it is going to be a very long time consuming process. You may or may not have to do much editing to your imported posts, it all comes down to just how picky you are.

Testing the Import

Now that the posts are all downloaded and in place it's time to test that the import worked as expected. Open up the terminal window and generate your site using the command below.
