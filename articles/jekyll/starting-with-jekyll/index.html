Starting wtih Jekyll

Before we Begin

Before we get started I need to make the disclaimer that I am not a Jekyll expert. The experiences outlined here are experiences I have had along the road to migrating to Jekyll. If you see something that is incorrect or if you know a better way to do something please speak up and let me know. I am providing this walk-through to get people started with migrating to Jekyll.

Is Jekyll Right for You?

Before you up and jump ship for Jekyll you need to make sure you are up for the task. Jekyll is not a one click solution like other blogging platforms out there, it is meant for developers who love to hack on things. It is going to take a considerable amount of effort to get the site layout in place and then you will find that many of the features you wish to have are not included so you have to start creating plugins so you can have this functionality.

Make sure you are up for the task before jumping in.

Installing Jekyll

Installing Jekyll requires that you already have Ruby and RubyGems intalled, I will not be covering that in this article. I am working on Mac OS X and it was already installed by default however I have installed homebrew and installed ruby manually so that I could be running on the latest version. To install jekyll execute the command I used below.

jcrawford@Workstation:~/jcrawford.github.com - article_plugin *% # gem install jekyll
Fetching: jekyll-0.12.0.gem (100%)
Successfully installed jekyll-0.12.0
1 gem installed
Installing ri documentation for jekyll-0.12.0...
Installing RDoc documentation for jekyll-0.12.0...
jcrawford@Workstation:~/jcrawford.github.com - article_plugin *% #

When you install it you might see other gems being installed as well, those are the required dependencies for jekyll and I already had them installed.

Setting Up Your Site

When you first start you will need to decide how you want to setup your site. You can choose one of two methods.

  1. Fork an existing site created by someone else
  2. Create your site from scratch.

I decided to go with creating my own from scratch because I didn't want to inherit anyone's layouts or custom plugins, I wanted this to be a learning experience. In this article I will be explaining how to create a site from scratch so if you fork a repository skip ahead to the next post in this series. I will also be using the Markdown format for my posts, however you are free to use whatever markup Jekyll supports. You can also see the sites entire source for this site in my Github Repository.

To start from scratch we need to create an initial directory structure with a few initial files. I have shown what your base directory should consist of but before you go and create this please read the explanations of what each are below.

Alternatively if you don't feel like typing all of the files out you can download an archive I created.

.
|-- \_config.yml
|-- \_includes
|-- \_layouts
|   |-- default.html
|   `-- post.html
|-- _posts
|   |-- 2012-12-21-my-first-post.md
|   |-- 2012-12-21-my-second-post.md
|-- \_site
`-- index.html
  • _config.yml
    This will store all of your site configuration data.
  • _includes
    This is where your partials will go so that you can reuse them in your layouts and templates.
  • _layouts
    This is where you will put your sites layout HTML files.
  • _posts
    This is where you will put all of your post files, each file MUST have a YAML Front Matter section as the first thing in the file. These files must also be named using the YEAR-MONTH-DAY-title.md.
  • _site
    When Jekyll builds your site this is the directory it uses to put the generated source in.

Creating Your Files

The first thing you will have to create is the \_config.yml file to store all of your site configurations.

title: "My Blog"
auto: false
server: false
lsi: false
permalink: /:year/:month/:title
markdown: rdiscount

The title section is a custom configuration which we will access in our layout files, this will allow us to set our site title in one spot and use it throughout our templates and layouts.

The auto setting is an option that you can pass to Jekyll through the command line, it tells Jekyll to remain running in it's loop watching for files that are changed in the sites source. When it detects a file has changed it will automatically run the build process again. I personally do not use this setting in my site and generate the site only when I wish to see the changes I have made.

The server setting tells Jekyll if it should run a basic HTTP server for you to view your site with, by default it loads up on port 4000. I do not use this as I have setup an apache vhost for testing my site but this is a very nice feature for those who do not have a web server setup.

The lsi setting tells Jekyll if it should create an index for related posts. I do not use this feature as it seems to take forever to generate the site, rather I use a plugin for getting related content by tag associations. This allows my site to generate rather quickly and still have related content shown on my site.

The permalink option allows you to specify not only how the post URL's will look but since the site is static you are also specifyin the generated sites directory structure. You will notice that it is using placeholders for the year, month and title. These will be replaced by Jekyll during the processing of the site with the actual year, month and title from the posts YAML Front Matter.

Jekyll supports a handful of markdown processors and you can choose whichever you want to use, everything I have read states that rdiscount is the fastest markdown processer currently out. I however have choosen to go with redcarpet so that I could use fenced code blocks like you do on Github rather than the highlight liquid tags. This support was recently added to Jekyll in the release 0.12.0 so if you are using 0.11 you will have to update or use a custom plugin to make use of the redcarpet parser.

Once you have your configuration file in place you need to create your default.html layout page. This will be the default layout that is used to render your pages. Create the file with the following contents.

<html>
  <head>
      <title>{{ site.title }}</title>
  </head>
  <body>
  {{ content }}
  </body>
</html>

As you can see this is a pretty basic file with very little markup at all. The important parts to note are the {{ tags }}. These will inject data into your layouts and templates. As you can see we are already pulling the site title from the configuration file we created. The other tag is for the actual content of the generated page, we will talk about that more shortly.

Go on and create the other layout file for your posts, this one can use the same contents as the layout.html for the time being having different layout files will allow you to have different layouts for your listing page and your view post page. For simplicity we will keep them the same, I just wanted to show you that you could have different layouts for different pages.

Next you should create your main index.html file which is located in the root of your project. This file is a bit different from the layouts file, this is the file that is used to generate the main index page of your blog so it will list your blog posts. We will again start off with a basic file.

<h1>My Posts</h1>
<ul class="posts">
{% for post in site.posts %}
  <li><span>{{ post.date | date: '%B %d, %Y' }}</span> &raquo; <a href="{{ post.url }}">{{ post.title }}</a></li>
{% endfor %}
</ul>

In this file you will notice that there is markup that is not HTML, this is called the Liquid Markup Language. Liquid is very useful for templates and layout files because it allows you to loop over collections of data during your site generation. In the above code we are simply looping over every blog post and adding it to an unordered list. We are listing it out by first printing a formatted date, don't worry if you don't understand the format specifiers. We are then creating a link to the post.url and giving it the title of post.title. You might notice that we are now accessing data from the post, these variables are setup by the Post object in Jekyll but you can also specify anything in the YAML Front Matter and access it through the post object. You will see how we will set the date in a few posts later.

Now that we have all of the required files put in place it is time to create a few test posts. Go ahead and create the 2012-12-31-my-first-post.md and 2012-12-31-my-second-post.md files in the _posts directory. Put the following contents in the files.

---
layout: post
date: 2012-12-31 13:00
---
## My First Post
This is my very first post.

---
layout: post
date: 2012-12-31 13:00
---
## My Second Post
This is my second post.

Once you have the files in place it's time to execute the build process, to do this you will have to go to terminal and make sure you are in your projects root directory. Once you are there execute the Jekyll command like I have shown below. 

jcrawford@Workstation:~/jcrawford.github.com - article_plugin *% # jekyll --server
Configuration from /Users/jcrawford/jcrawford.github.com/_config.yml
Building site: /Users/jcrawford/jcrawford.github.com -> /Users/jcrawford/jcrawford.github.com/_site
Successfully generated site: /Users/jcrawford/jcrawford.github.com -> /Users/jcrawford/jcrawford.github.com/_site
[2012-12-31 22:39:18] INFO  WEBrick 1.3.1
[2012-12-31 22:39:18] INFO  ruby 1.9.3 (2012-11-10) [x86_64-darwin11.4.2]
[2012-12-31 22:39:18] INFO  WEBrick::HTTPServer#start: pid=36970 port=4000

You should see similar results that I have shown above, if not then something must have gone wrong with your setup of the files or with your actual installation of Jekyll. You will also notice that I used the --server argument when running the Jekyll command this tells Jekyll to fire up a basic web server using the WEBrick ruby gem so that you can test your site. You can now load up your browser and type in http://localhost:4000/ and you should see your site. As mentioned it is VERY basic but you should see an unordered list with the 2 posts we created earlier. You should also be able to click on the links and it should take you to view the contents of the post.
